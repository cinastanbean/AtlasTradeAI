# 智能能力分析与 Agent 演进设计

## 1. 文档目的

本文档用于回答两个关键问题：

1. AtlasTradeAI 未来作为一个可以独立运转的智能贸易操作系统，它的“智能”到底体现在哪里。
2. 当前系统后续还需要补哪些 Agent，这些 Agent 分别应该由规则引擎、大模型，还是二者混合驱动。

## 2. 这个系统的智能不只是“会聊天”

AtlasTradeAI 的智能，不应该理解成一个会对话的机器人，而应该理解成以下四类能力的组合：

### 2.1 感知智能

系统能够感知业务中发生了什么。

例如：

- 识别某张订单进入了新的生命周期阶段
- 识别生产是否延期
- 识别单证是否缺失
- 识别物流是否偏离承诺时效
- 识别回款是否临期或逾期
- 识别客户是否处于投诉、流失或高价值维护阶段

这一层的本质，是把多系统数据转成统一业务事件。

### 2.2 判断智能

系统能够对当前事件做风险判断和优先级判断。

例如：

- 生产延期是否足以影响交付
- 单证缺失是否会阻塞发货
- 物流延误是否需要主动通知客户
- 回款逾期是否需要升级财务风险
- 客户投诉是否会影响复购或客户等级

这一层通常应由规则引擎打底，以保证稳定、可审计。

### 2.3 编排智能

系统能够决定下一步由谁接管、做什么、何时完成。

例如：

- 订单确认后自动触发 Follow-up Agent
- 单证缺失后同时触发 Customs Agent 和 Follow-up Agent
- 回款逾期后同时触发 Finance Agent 和 Follow-up Agent
- 售后投诉后触发 Customer Service Agent 和 CRM Agent

这一层是“智能操作系统”的真正骨架，因为它体现的是自动流转，而不是手工点选。

在当前工程里，这一层已经由 `Order Orchestrator` 开始承接。

### 2.4 表达智能

系统能够用更接近业务人员的语言输出摘要、建议、通知和沟通草稿。

例如：

- 自动生成适合钉钉通知的提醒文案
- 自动生成销售沟通建议
- 自动生成售后回复要点
- 自动生成经营分析摘要

这一层更适合由大模型增强，但不能替代规则和流程控制。

## 3. 这个系统的智能体现在哪些地方

如果从产品和运行层面来看，AtlasTradeAI 的智能主要体现在以下 6 个点：

### 3.1 订单不是静态记录，而是主动驱动流程的中枢对象

订单一旦进入系统，就不再只是一个被查询的数据对象，而是成为：

- 事件订阅的主对象
- 状态机流转的主对象
- 多 Agent 协同的主对象
- 风险和任务聚合的主对象

### 3.2 系统能够主动发现异常，而不是等待人工发现

例如：

- 系统主动发现某订单生产延期
- 系统主动发现某订单单证不完整
- 系统主动发现某订单已逾期未回款

### 3.3 系统能够自动生成下一步动作

例如：

- 生成任务
- 生成异常
- 生成通知
- 生成责任归属
- 生成下一阶段的接管 Agent

### 3.4 系统能够让不同角色 Agent 在同一事件链上协同

例如：

- Follow-up Agent 负责推进
- Supply Chain Agent 负责执行条件校验
- Logistics Agent 负责时效判断
- Finance Agent 负责资金风险识别

### 3.5 系统能够沉淀“经营记忆”

例如：

- CRM Agent 沉淀客户经营变化
- Customer Service Agent 沉淀售后与投诉闭环经验
- Finance Agent 沉淀客户账期和回款习惯

### 3.6 系统能够对人输出更高质量的建议和表达

这正是大模型最适合介入的位置。

## 4. 后续还需要哪些 Agent

当前系统已经具备：

- Sales Agent
- CRM Agent
- Follow-up Agent
- Supply Chain Agent
- Logistics Agent
- Customs / Documentation Agent
- Finance Agent
- Customer Service Agent

但从长期看，后续还可以继续扩展为以下几类：

### 4.1 Order Orchestrator

这不是普通 Agent，而是订单中枢的编排智能层。

职责：

- 根据事件决定下一步触发链
- 维护订单生命周期状态机
- 统一路由不同角色 Agent

### 4.2 Document Intelligence Agent

当前 Customs Agent 更偏流程与阻塞判断。
后续可以进一步拆出单证智能 Agent，专门负责：

- 单证抽取
- 单证比对
- 报关资料一致性审查
- 模板文档草稿生成

### 4.3 Operations Analyst Agent

负责经营分析、日报周报、问题诊断。

职责：

- 分析延期率、交付率、回款率
- 分析高风险客户与订单
- 输出经营报告摘要

### 4.4 Knowledge Agent

负责知识沉淀与检索。

职责：

- 沉淀 SOP
- 沉淀历史异常处理案例
- 为其他 Agent 提供知识检索支持

## 5. 哪些 Agent 适合规则引擎

以下 Agent 更适合以规则引擎为主：

- Supply Chain Agent
- Customs / Documentation Agent
- Order Orchestrator

原因：

- 这些 Agent 面对的是强结构化流程
- 需要明确的状态边界
- 需要可审计、可回放、可人工接管

## 6. 哪些 Agent 适合大模型增强

以下 Agent 最适合大模型增强：

- Sales Agent
- CRM Agent
- Follow-up Agent
- Logistics Agent
- Finance Agent
- Customer Service Agent
- Operations Analyst Agent

原因：

- 这些 Agent 需要较强的语言表达能力
- 需要总结、解释、建议、沟通
- 需要面向人输出更自然的结论

## 7. 哪些 Agent 最适合“规则 + 大模型混合”

真正可落地的方式，不是纯规则，也不是纯大模型，而是混合模式。

建议如下：

### 7.1 Follow-up Agent

- 规则负责识别风险和生成任务结构
- 大模型负责优化摘要、建议和通知

### 7.2 Finance Agent

- 规则负责识别逾期、账期和风险等级
- 大模型负责生成催收建议和财务沟通摘要

### 7.3 CRM Agent

- 规则负责客户分层和提醒条件
- 大模型负责客户画像总结和经营建议

### 7.4 Sales Agent

- 规则负责成交推进节点和动作模板
- 大模型负责销售沟通建议与报价表达

### 7.5 Customer Service Agent

- 规则负责投诉分流和 SLA 判断
- 大模型负责回复建议、问题摘要和闭环复盘

## 8. 当前工程已经实现到哪一步

当前代码工程已经具备以下智能形态：

- Follow-up Agent 已支持规则 + 大模型混合
- Sales Agent 已具备成交推进判断逻辑
- CRM Agent 已具备客户经营沉淀判断逻辑
- Supply Chain Agent 已具备供应链风险判断逻辑
- Logistics Agent 已具备交付时效判断逻辑
- Customs Agent 已具备单证阻塞判断逻辑
- Finance Agent 已具备回款风险判断逻辑
- Customer Service Agent 已具备投诉处理判断逻辑

这些 Agent 当前已经可以基于事件被真实触发，而不是只停留在文档设计层。

## 9. 结论

AtlasTradeAI 的智能，核心不是“某个 Agent 很聪明”，而是：

- 系统能主动感知业务变化
- 系统能稳定做出业务判断
- 系统能自动编排流程流转
- 系统能把多角色 Agent 串成同一条事件链
- 系统能用更适合业务人员的方式输出结论

因此，这个项目的正确方向不是不断叠加聊天能力，而是持续增强：

- 事件感知能力
- 规则判断能力
- 流程编排能力
- 多 Agent 协同能力
- 大模型表达增强能力
